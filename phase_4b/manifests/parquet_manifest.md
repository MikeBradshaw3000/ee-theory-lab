# Parquet Manifest — Schema Documentation

**Document role.** Canonical schema specification for the parquet manifest at `phase_4b/manifests/parquet_manifest.csv`. The manifest is the per-file verification and identity artifact for the eight Flight 6 production parquet files at `flight2_outputs/`, plus any future parquet outputs covered by the same discipline.

**Authority.** Authoritative for manifest schema (field names, allowed values, semantics) and for the manifest-update flow (script-driven regeneration, not manual maintenance). Not authoritative for the substrate's content or for verification rule definitions (those live in `Flight 6 Substrate Specification v1.1 §14.1` and in `phase_4b/scripts/_manifest_verification.py`).

**Maintenance discipline.**
- Schema changes are versioned (`manifest_schema_version` field). Schema version increments commit alongside this documentation update.
- Manifest data is regenerated by `phase_4b/scripts/regenerate_manifest.py`. Operations logs record when the script ran, its summary output, and whether the manifest changed.
- The manifest is a generated verification artifact, not a manually maintained table. Direct edits to the CSV are prohibited; per Rule 7, the discipline against inference-based repair applies — if regeneration produces unexpected results, the diagnosis happens at the substrate level or the script level, not by hand-editing manifest rows.

**Relationship to other documents.**
- `Flight 6 Substrate Specification v1.1 §14.1` — verification structure source (file existence, file size, row count, column count, required columns, tick range, unique cells, F_variant, non-empty, realization invariant, clipping summary).
- `Flight 6 Substrate Specification v1.1 §13.2` — shadow-copy structure source (F_2_symmetric runs produce one underlying parquet file plus three probe-named files via byte-identical shadow copies; F_LR runs produce one parquet file each).
- `protocols/foundational/canonical_artifacts_index.md` Section 5 — substrate data path and naming notes.
- `protocols/foundational/standing_rules.md` Rule 7 — no inference-based repair of missing required telemetry; the manifest's non-repair discipline derives from Rule 7.
- ChatGPT Q4 recommendation (session 6) — manifest-now-Git-LFS-later storage philosophy and the `logical_artifact_id` / `shadow_copy_group` / `byte_identical_to` identity-structure fields.

---

## Schema synthesis

The schema combines two source frames:

- **Identity & shadow-copy structure** (ChatGPT Q4 + FSS v1.1 §13.2): what this physical file is, which logical artifact it represents, and how it relates to other physical files.
- **Per-file verification payload** (FSS v1.1 §14.1): whether the file satisfies substrate-output expectations.

The row unit is **the physical parquet file**. Shadow-copy fields connect multiple physical files into one logical artifact group; verification fields establish the file's health.

## Schema, version 1.0

### Identity & structure fields

| Field | Type | Description |
|---|---|---|
| `logical_artifact_id` | string | Canonical identifier for the underlying genuine substrate run that this physical file represents. Multiple physical files (shadow copies) may share one `logical_artifact_id`. |
| `physical_file_path` | string | Absolute filesystem path to the physical parquet file. |
| `artifact_role` | string | What role this physical file plays — e.g., `production`, `shadow_copy`, `archive`. |
| `scale` | string | Substrate grid scale, e.g., `20x20`, `40x40`. |
| `F_variant` | string | F-form variant — `F_2_symmetric` or `F_LR`. |
| `probe_name` | string | Probe identifier — e.g., `probe1_overcrowding`, `probe2_starvation_F2sym`, `probe2_starvation_FLR`, `probe3_fusion_residual`. |
| `shadow_copy_group` | string | Group identifier connecting physical files that are byte-identical or are claimed to be byte-identical per FSS §13.2. Files in the same group share a `logical_artifact_id`. |
| `byte_identical_to` | string | If this file is a shadow copy, the `physical_file_path` of the primary file. Empty for the primary file of a group. |
| `shadow_copy_status` | enum | One of: `genuine`, `verified_shadow_copy`, `presumed_shadow_copy`, `not_applicable`. |
| `byte_identity_verification_method` | enum | One of: `sha256_match`, `file_size_and_spec_only`, `not_checked`, `not_applicable`. This field is intentionally a compact method/result enum — `sha256_match` reads as "method: SHA-256 hash comparison; result: match." A future schema revision could split into separate method and result fields if that distinction needs to be tracked independently; the current design treats the verification outcome and the method as one paired record because byte-identity verification is binary at the per-file level (a single SHA-256 comparison is either a match or it isn't). |
| `byte_identity_verified_at` | timestamp | When the byte-identity check ran. Empty when `byte_identity_verification_method` is `not_checked` or `not_applicable`. |
| `byte_identity_verified_by` | string | Identifier of the entity that performed the byte-identity check (typically `regenerate_manifest.py` or a manual session reference like `session_6_sha256`). |

**`shadow_copy_status` semantics.**
- `genuine` — this file is the underlying substrate run, not a copy of another file.
- `verified_shadow_copy` — this file has been independently confirmed (e.g., via SHA-256 hash match) to be byte-identical to another file in the same `shadow_copy_group`.
- `presumed_shadow_copy` — this file is claimed by FSS §13.2 specification to be a shadow copy but has not been independently verified.
- `not_applicable` — the shadow-copy framing does not apply (e.g., F_LR files which FSS §13.2 specifies as having one parquet file each).

The distinction between `verified_shadow_copy` and `presumed_shadow_copy` is load-bearing. Future readers consulting the manifest should be able to distinguish "SHA-256-confirmed byte-identity" from "expected-by-spec byte-identity" without re-running the verification step.

### Verification payload fields (per FSS v1.1 §14.1)

| Field | Type | Description |
|---|---|---|
| `file_exists` | bool | Whether the file exists at `physical_file_path` at manifest-generation time. |
| `file_size_bytes` | int | Filesystem-reported size in bytes. |
| `sha256` | string | SHA-256 hash of file contents computed by the regenerate script at every regeneration. This is the file's identity hash — not the same operation as byte-identity verification (which compares SHA-256 hashes across files in a `shadow_copy_group` to upgrade `shadow_copy_status` from `presumed` to `verified`). The `sha256` field is populated at every manifest regeneration; the byte-identity verification operation is separate (see verification-vs-generation separation below). |
| `row_count` | int | Total parquet rows. |
| `column_count` | int | Total parquet columns. |
| `required_columns_present` | bool | Whether all required columns (per FSS v1.1 §14.1) are present. |
| `missing_required_columns` | string | Comma-separated list of missing required columns; empty when all present. |
| `tick_min` | int | Minimum tick value. |
| `tick_max` | int | Maximum tick value. |
| `tick_count` | int | Number of distinct ticks. |
| `unique_cells` | int | Number of distinct cell identifiers. |
| `expected_cells` | int | Expected cell count based on `scale` (e.g., 400 for 20x20, 1600 for 40x40). |
| `non_empty` | bool | Whether the file contains any rows. |
| `realization_invariant_satisfied` | bool | Whether the realization invariant per FSS v1.1 §14.1 is satisfied. |
| `clipping_summary` | string | Summary string capturing clipping diagnostics per FSS §14.1 (specific format defined in `phase_4b/scripts/_manifest_verification.py`). |
| `verification_status` | enum | One of: `pass`, `fail`, `warning`, `not_checked`. |
| `verification_notes` | string | Free-form notes from the verification step. For `fail` or `warning` status, this field carries the specific reason. |

**`verification_status` semantics.**
- `pass` — all per-FSS §14.1 checks satisfied.
- `fail` — one or more required checks failed (e.g., missing required columns, wrong row count for scale, realization invariant violated).
- `warning` — non-blocking concern surfaced (e.g., row count is within tolerance but at the edge of expected range).
- `not_checked` — verification did not run (either because verification was scoped out or because the file did not exist).

Per Rule 7 (no inference-based repair): the script must not silently repair missing or malformed parquet outputs. It produces manifest rows with `fail` status and explicit `verification_notes`. Diagnosis happens at the substrate level, not by manifest editing.

### Provenance fields

| Field | Type | Description |
|---|---|---|
| `manifest_schema_version` | string | Schema version of this manifest (currently `1.0`). |
| `manifest_generated_at` | timestamp | ISO 8601 timestamp of manifest generation. |
| `manifest_generated_by_script` | string | Path to the script that generated the manifest (typically `phase_4b/scripts/regenerate_manifest.py`). |
| `source_data_root` | string | Absolute path to the parquet data root inspected by the script (typically `C:\Users\vkz244\EE_Theory_Lab\flight2_outputs\`). |

## Row ordering

The script produces rows in deterministic order to reduce meaningless diffs:

```text
ORDER BY scale ASC, F_variant ASC, probe_name ASC, physical_file_path ASC
```

This ordering is enforced by `regenerate_manifest.py`. Manual edits that reorder rows will be reverted by the next regeneration.

## Verification-vs-generation separation

The manifest is generated by `regenerate_manifest.py`. Verification logic — the per-FSS §14.1 checks (`required_columns_present`, `realization_invariant_satisfied`, `clipping_summary`, etc.) — lives in a separate module at `phase_4b/scripts/_manifest_verification.py`.

This separation matches the existing `_phase_4b_intake.py` (reusable module) + `tier3_regression.py` (consumer script) pattern. A future verification consumer (e.g., a CI-style check script) can import `_manifest_verification.py` directly and run the same checks against arbitrary parquet inputs without invoking manifest generation.

**Three operations, distinguishable in the schema:**

1. **Manifest regeneration** — computes per-file fields including `sha256` (the file's identity hash). Runs whenever parquet outputs change. Populates all schema fields.

2. **Substrate verification** — applies FSS §14.1 checks via `_manifest_verification.py`. Populates `verification_status`, `verification_notes`, and the §14.1 payload fields. Runs as part of manifest regeneration.

3. **Byte-identity verification** — compares SHA-256 hashes across files in a `shadow_copy_group` to upgrade `shadow_copy_status` from `presumed_shadow_copy` to `verified_shadow_copy`. Separate from manifest regeneration. Updates the verification-record store at `phase_4b/manifests/byte_identity_verifications.csv`; subsequent manifest regeneration consults the store and reflects the updated status. Runs at session-level discretion (not at every regeneration), preserving the honest-record distinction between expected-by-spec and verified-by-hash.

The `sha256` field is populated at every manifest regeneration (operation 1). Byte-identity verification (operation 3) uses those hashes but is not the same operation as computing them.

## Initial population scope

Per Mike's Q2b arbitration: Stage 3 produces schema specification only. Manifest population (running `regenerate_manifest.py` against the production parquets) is deferred to Layer 3 routing.

The first population will produce manifest rows for the eight Flight 6 production parquet files:

- 20x20 scale, four probe-name variants (`probe1_overcrowding`, `probe2_starvation_F2sym`, `probe2_starvation_FLR`, `probe3_fusion_residual`)
- 40x40 scale, four probe-name variants (same as above)

Per session 6 SHA-256 verification, the 20x20 `shadow_copy_status` distribution is known: `probe1_overcrowding` is `genuine`; `probe2_starvation_F2sym` and `probe3_fusion_residual` are `verified_shadow_copy`; `probe2_starvation_FLR` is `not_applicable` (F_LR variant, not in a shadow-copy group per FSS §13.2). The 40x40 distribution is `presumed_shadow_copy` for the corresponding F_2_symmetric files until SHA-256 verification establishes the pattern at the 40x40 scale.

The initial population should record these statuses honestly: `verified_shadow_copy` only where SHA-256 verification has been performed; `presumed_shadow_copy` elsewhere. A subsequent verification session can run SHA-256 across the 40x40 files and update the manifest from `presumed` to `verified`.

## Example row note

The single example row in `parquet_manifest.csv` is illustrative-only, not generated by the script. It demonstrates field meanings using session-6-verified data for `probe1_overcrowding_20x20`. Placeholder values (`EXAMPLE_64_HEX_SHA256`, `EXAMPLE_FILE_SIZE`, `EXAMPLE_ROW_COUNT`) are marked as such to prevent later readers from mistaking them for real data. The `git_head_at_generation` value in the example row is `not_available` matching the documented runtime fallback semantics; in practice the script populates this with the actual git HEAD SHA at generation time (or `not_available` if git metadata cannot be read).
