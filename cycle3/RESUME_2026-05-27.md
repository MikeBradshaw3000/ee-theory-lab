## Carry-forward cautions (NEW from this session, plus prior carries)

### New from this session

- **Silent Move-Item batching loses trace.** A multi-file batch loop that emits "moved" messages on `if ($newest)` is reporting source-file existence, not Move-Item success. If the destination directory doesn't exist, Move-Item raises errors and the "moved" log lies. Use `try { Move-Item -ErrorAction Stop; "OK: ..." } catch { "FAILED: ..." }` for batches. Caught this session via stderr inspection; recovery was a one-line directory creation.

- **BOM and trailing-blank-line conventions are per-file, not project-wide.** The anchor says "canonical docs here are BOM-less UTF-8" but the registry was an exception (BOM-prefixed with trailing blank line). Verify both first-3-bytes AND ending-bytes against the actual on-disk file before generating replacements. The discipline is "match the original," not "normalize to convention."

- **Multi-AI routing pattern worked cleanly this session - positive precedent.** Layer 1 contract -> Layer 3 implementation -> Mike's canonical run -> Layer 3 interpretation -> Layer 1 implementation review (script content) + sufficiency-test (predicted output vs actual CSV) -> Layer 2 substantive review -> Layer 1 synthesis to canonical record. Framing-asymmetry guard held: Layer 2 walked back the "routing a correction" framing on independent grounds on the CTL-001 question, then accepted SS-001 on independent grounds on the closeout question. The contract's pessimistic-on-passing language was effective.

- **Layer 3 declared L2 clearance in their own voice on first response ("L2 Sufficiency Check Passed").** The contract's anticipation of this category error was effective; the framing was reframed at Layer 1's pushback into predicted-output language.
